1 linux alsr地址随机化

1.1 遇到的问题

在一次dpdk应用开发中，使用的多进程模型。其中在master进程启动时，使用rte_hash_create创建了hash表。 

/* Parameters used for task hash table */
static struct rte_hash_parameters task_params = {
	.name = "pf_task_hash_table",
	.entries = 65535,
	.key_len = sizeof(uint32_t),
	.hash_func = rte_jhash,
	.hash_func_init_val = 0,
	.socket_id = 0,
};

....

task_hash = rte_hash_create(&task_params);

其中初始化了计算hash函数（hash_func），保存的是函数地址，等子进程启动后，进行hash添加操作时，在计算hash时，其地址无效，导致Segmentation Fault。经过调试，在master进程中和slave进程中，hash_func地址相同，但是slave中此地址不是函数rte_jhash的地址。经过一番搜索定位到了linux alsr地址随机化技术。

1.2 解决

查看当前操作系统的ASLR配置情况： 

$ cat /proc/sys/kernel/randomize_va_space
2
#或者使用
$ sudo sysctl -a  -r randomize_va_space
kernel.randomize_va_space = 2

randomize_va_space值的含义： 0 = 关闭 1 = 半随机。共享库、栈、mmap() 以及 VDSO 将被随机化。 2 = 全随机。

关闭alsr的命令： 

# echo 0 > /proc/sys/kernel/randomize_va_space
//或者使用
# sysctl -w kernel.randomize_va_space=0

然后在启动应用时，一切正常了。

2 ioremap()、ioremap_cache()、memremap()

在一次开发中，需要预留出系统的物理内存，通过驱动映射至用户空间。

预留物理内存的方法：

编辑`/etc/default/grub`文件，在`GRUB_CMDLINE_LINUX=""`中添加需要保留的内存大小和内存起始地址 ! 例如： GRUB_CMDLINE_LINUX="memmap=8G\$0x1080000000 memmap=8G\$0x1280000000"

  • 参数 8G为预留内存大小，单位为[K,M,G]
  • 0x1080000000为预留内存起始地址

更新grub，重启系统 sudo update-grub && sudo reboot

之后通过编写的驱动程序将此物理内存映射至用户空间后，发现读写此内存很慢，代码中是使用函数ioremap（）来映射的此内存，经过查看内核源码： 

/*
 * ioremap() and friends.
 *
 * ioremap() takes a resource address, and size.  Due to the ARM memory
 * types, it is important to use the correct ioremap() function as each
 * mapping has specific properties.
 *
 * Function             Memory type     Cacheability    Cache hint
 * ioremap()            Device          n/a             n/a
 * ioremap_cache()      Normal          Writeback       Read allocate
 * ioremap_wc()         Normal          Non-cacheable   n/a
 * ioremap_wt()         Normal          Non-cacheable   n/a
 *
 * All device mappings have the following properties:
 * - no access speculation
 * - no repetition (eg, on return from an exception)
 * - number, order and size of accesses are maintained
 * - unaligned accesses are "unpredictable"
 * - writes may be delayed before they hit the endpoint device
 *
 * All normal memory mappings have the following properties:
 * - reads can be repeated with no side effects
 * - repeated reads return the last value written
 * - reads can fetch additional locations without side effects
 * - writes can be repeated (in certain cases) with no side effects
 * - writes can be merged before accessing the target
 * - unaligned accesses can be supported
 * - ordering is not guaranteed without explicit dependencies or barrier
 *   instructions
 * - writes may be delayed before they hit the endpoint memory
 *
 * The cache hint is only a performance hint: CPUs may alias these hints.
 * Eg, a CPU not implementing read allocate but implementing write allocate
 * will provide a write allocate mapping instead.
 */
void __iomem *ioremap(resource_size_t res_cookie, size_t size);
#define ioremap ioremap

/*
 * Do not use ioremap_cache for mapping memory. Use memremap instead.
 */
void __iomem *ioremap_cache(resource_size_t res_cookie, size_t size);
#define ioremap_cache ioremap_cache

发现ioremap一般用来映射设备的I/O空间，而且缓存对其没有作用，所以导致内存的读写很慢。

之后使用memremap函数（内核建议使用memremap代替ioremap_cache函数）进行映射，读写速度大大提升。
