1. 向量包处理

1.1. 标量包处理

使用标量包处理时，网络栈通常每次处理一个包，中断处理函数从网卡获取一个包，并将包传递给一系列函数进行处理。 

+---> fooA(packet1) +---> fooB(packet1) +---> fooC(packet1)
+---> fooA(packet2) +---> fooB(packet2) +---> fooC(packet2)
...
+---> fooA(packet3) +---> fooB(packet3) +---> fooC(packet3)

标量包处理过程比较简单，易于使用，但效率较低，例如：

  • 处理包过程中，当代码路径的长度超过处理器的指令缓存（I-cache），由于处理器不停的加载新的指令导致出现颠簸现象thrashing。这种情况下，每个数据包都会产生相同集合的 I-cache MISS。
  • 调用栈较深时，栈中内容会从处理器一级数据缓存（D-cache）中挤出，从而会增加LSU（load-store-uint）压力。

1.2. 向量包处理

使用向量包处理时，一次处理多个数据包，称作“包向量”或者向量。中断处理函数一次从网卡获取包向量（多个数据包），并将向量传递给处理函数进行处理。 

+---> fooA([packet1, +---> fooB([packet1, +---> fooC([packet1, +--->
            packet2,             packet2,             packet2,
            ...                  ...                  ...
            packet256])          packet256])          packet256])

这种方法可以解决标量包处理中问题：

  • 多个包分摊 I_cache 的负载，解决了I-cache颠簸问题（thrashing）。
  • 单次从网卡收取最大256的包向量，处理时，直接调用有向图的节点处理。有向图调度器 每次调用一个节点的发送函数，从而将调用栈的深度限定在较小的栈帧范围。

这种方式可以进一步优化：开启 pipelining 和 prefetching（预取），最小化表数据的读取延迟，以及并行处理数据包的负载。

2. Packet Processing Graph

VPP 设计的核心是 Packet Processing Graph ，具有以下优点：

  • 可插拔，易于理解和扩展
  • graph node 架构成熟
  • pipeline过程可任意编排

VPP 中，数据包处理过程分解到 prcket processing graph 中，任何人都可以将插入新的 graph node，因此可以通过插件实现特殊的需求。

在运行过程中，VPP 从RX ring中获取 包向量，通常单个包向量最大为256。然后将包向量交给 graph，挨个执行 node 来处理包向量。

Graph nodes 实现小巧，模块化，node和node之间关联性低，因此可以很容易引入新的node，改写 graph node 也很方便。

插件使用 动态库 来实现，vpp启动时通过查找插件目录，动态加载插件。插件可以引入新的 graph node ，或者重新编排 packet processing graph 。

    • Arbitrary n-tuple classifiers

  • Network and cryptographic hardware support with DPDK.
  • Container and Virtualization support
    • Para-virtualized interfaces; Vhost and Virtio
    • Network Adapters over PCI passthrough
    • Native container interfaces; MemIF

  • Universal Data Plane: one code base, for many use cases
    • Discrete appliances; such as Routers and Switches.
    • Cloud Infrastructure and Virtual Network Functions
    • Cloud Native Infrastructure
    • The same binary package for all use cases.
  • Out of the box production quality, with thanks to CSIT.

4. Host Stack

依赖于 VPP 基于 graph 的转发模型，以及 vector packet processing，使得VPP Host Stack具有高吞吐。借助 host stack 开放的API，用户应用程序可以高效的读取以及发送数据，同时也为 内部app 提供了高效的通信机制。

从上层看，VPP Host stack 主要包含3部分内容：

  • session 层： 用于促进 传输层协议和应用层 之间的交互
  • 可插拔的传输层协议： TCP QUIC TLS UDP
  • VCL (VPPComs library) ：一组通用库，目的是使应用与stack交互更加方便

所有组件都是定制的，目的是为了适配VPP的架构，并能充分利用VPP优势。因此在以下方面投入了大量精力：

  • 抽象了一套可插拔的会话层，应用程序和传输层之间通过定制的共享内存组件交互。 有了会话层的存在，就能够在VPP中实现 QUIC、TLS 协议。
  • TCP实现：支持向量包处理，并且延续了 VPP 的高度扩展的线程模型。遵循了RFC规范，支持高版本的 high-speed TCP 协议特性，并通过了 Defensic’s Codenomicon 1M+ tests suite 认证。
  • VCL：在用户空间实现的类似传统异步通信的库。
  • 实现了高性能的 “cut-through” 通信机制： 允许应用关联到VPP，通过共享内存与VPP交换数据，并且不存在传统传输协议的带来的开销。通过测试，这种机制比传统的容器间网络传输更加高效。

5. 更多特性

本小节介绍一些 VPP环境相关内容，以及开发者可以使用的特性。

  • Extensive runtime counters; throughput, intructions per cycle, errors, events etc.
  • Integrated pipeline tracing facilities
  • Multi-language API bindings
  • Integrated command line for debugging

  • Fault-tolerant and upgradable

    • Runs as a standard user-space process for fault tolerance, software crashes seldom require more than a process restart.
    • Improved fault-tolerance and upgradability when compared to running similar packet processing in the

    kernel, software updates never require system reboots.

    • Development experience is easier compared to similar kernel code
    • Hardware isolation and protection (iommu)
  • Built for security
    • Extensive white-box testing
    • Image segment base address randomization
    • Shared-memory segment base address randomization
    • Stack bounds checking
    • Static analysis with Coverity
