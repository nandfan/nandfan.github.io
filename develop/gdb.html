1 Linux Coredump文件

coredump文件时进程崩溃时产生的磁盘文件，包含了进程崩溃时内存的映像，该文件可以用调试器（例如gdb）来调试，用于查看进程崩溃时的状态信息。

1.1 coredump文件命名

默认情况下coredump文件名为core，查询coredump文件路径： 

# cat /proc/sys/kernel/core_pattern
core
# sysctl kernel.core_pattern
kernel.core_partern = core

coredump文件模板配置：

%%
单个%字符
%c
进程崩溃时的文件大小软限制
%e
进程或者线程的comm值，通常是可执行文件的文件名（不包括路径，最长为15个字符），该值可能被修改（/proc/pid/comm /proc/pid/task/tid/comm）
%E
可执行文件的路径名
%g
转储进程的GID
%h
主机名（hostname）
%i
触发核心转储的TID(线程所在PID命名空间内部）
%I
触发核心转储的TID(初始PID命名空间）
%p
进程的PID（进程所在PID命名空间）
%P
进程的PID（初始PID命名空间）
%s
引发转储的信号
%t
转储的时间戳
%u
进程的UID

配置coredump文件模板示例： 

# echo "/var/log/core_%e_%p" > /proc/sys/kernel/core_pattern
或者使用
# sysctl -w kernel.core_patterm=/var/log/core_%e_%p

要产生core文件，还需要设置RLIMIT_CORE限制，如果为0则不产生core文件。 

# ulimit -c
0
# ulimit -c core文件大小限制      //设置core文件大小限制
# ulimit -c unlimited             //不限制core文件大小

2 gdb

2.1 指定从文件读取gdb命令

# gdb --command=gdb-command-file -p pid

–command可以用来从文件读取gdb配置命令，在文件中配置断点或者其他调试信息。 以下是一个快速调试nginx进程的示例: 

# gdb nginx --command=nginx.gdb

# cat nginx.gdb
#set enviroment NAME=VALUE可以用来设置进程的环境变量
#set arg 可以用来设置进程启动参数
set follow-fork-mode child  # 设置跟踪子进程

b ngx_event_pipe_write_to_downstream   #断点
b ngx_http_upstream_process_downstream
b ngx_http_upstream_process_upstream

快速调试进程的脚本，（以下用nginx worker进程示例）： 

#! /bin/bash

ngx_pid=`ps axjf | grep "nginx: worker process" | grep -m 1 -v "grep" | awk '{ print $2 }'`
gdb -p $ngx_pid --command=nginx.gdb
